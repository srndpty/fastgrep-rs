# AGENTS.md

このリポジトリで作業する人間・AIエージェント向けのガイドラインです。

## プロジェクト概要

`fastgrep-rs` は、ローカルのコード・Markdown・法令テキスト・ログを高速に検索する
CLI ツールです。`ignore` でディレクトリを再帰走査し、`regex` で行単位検索して
`path:line_number:line` 形式で出力します。単一バイナリ配布（特に Windows）を重視します。

将来的な用途: 自作 grep / ripgrep ライクツール、検索アプリのインデクサ、
Python 製ツールの高速化部品。

## 言語ポリシー（最優先）

**チャットの返答、コードのコメント、コミットメッセージ、レビューコメントなど、
コード本体以外で人間が読むことを想定している文章は原則として日本語にすること。**

- 対象: 会話の応答、ドキュメント（本ファイル等）、コード内コメント、doc コメント、
  コミットメッセージ、PR の説明、レビューコメント、`eprintln!` 等のログ文言の検討。
- 例外: 識別子（変数・関数・型名）、コードそのもの、外部 API や規約で英語が必須の箇所、
  既存の英語慣用句（例: `TODO`, `FIXME`）。CLI の `--help` 文言は日本語で問題ない。
- 技術用語は無理に訳さず、一般的なカタカナ/英語表記で構わない（例: ストリーミング、BufWriter）。

## 開発フロー

変更後は以下を**すべて**通すこと（CI 相当の最低ライン）。

```sh
cargo fmt                      # 整形（コミット前に必ず実行）
cargo clippy -- -D warnings    # lint。警告ゼロを必須とする
cargo test                     # ユニットテスト
```

動作確認・配布:

```sh
cargo run -- "<pattern>" <path>     # 開発中の即時確認
cargo build --release               # target/release/fastgrep-rs(.exe)
cargo install --path . --force      # ~/.cargo/bin に配置（更新時は --force）
```

注意: `cargo install` 済みバイナリはその時点のスナップショット。コードを変えたら
再 `cargo install` しないと反映されない。開発中の確認は `cargo run` を使う。

## コーディング規約（Rust）

- **edition 2024**。`rustfmt` 既定スタイルに従う（独自整形をしない）。
- `clippy -- -D warnings` を常にクリーンに保つ。let-chain など stable な新構文は使ってよい。
- エラー処理は `anyhow`（`Context`/`with_context` で文脈を付与）。
  - アプリ境界では `Result` を返し `?` で伝播。`main` も `-> anyhow::Result<()>`。
  - 本番コードで `unwrap()`/`expect()` を安易に使わない（テストや不変条件が自明な箇所は可）。
- I/O はストリーミングを基本とし、ファイル全体や全マッチをメモリに溜め込まない
  （巨大ファイル・大量ファイルでの安定性が本ツールの肝）。
- 出力（整形・色）と検索ロジックを混在させない。検索は値/コールバックで結果を渡し、
  表示は呼び出し側（`output` モジュール）に委ねる。
- 文字列はバイト幅ではなく文字（`char`）境界を壊さないよう扱う（多言語テキスト対応）。

## モジュール構成

| ファイル | 責務 |
| --- | --- |
| `src/main.rs` | CLI 定義（clap）と全体の配線（`run`/`main`）。色・バッファリング・走査の制御。 |
| `src/search.rs` | ファイル走査・ストリーミング読み取り・行マッチング。`read_capped_line`/`search_reader`/`search_file`/`SearchError`。 |
| `src/output.rs` | 整形と色付き出力。`split_visible`/`Printer`（パス・行番号・一致箇所の強調）。 |
| `src/size.rs` | サイズ文字列（`10M`/`500K`/`1M` 等）のパース。 |
| `tests/cli.rs` | `assert_cmd` による CLI 統合テスト（出力フォーマット・終了コード・各オプション・逐次/並列の一致など）。 |

各モジュールは末尾に `#[cfg(test)] mod tests` を持ち、ユニットテストを同居させる。
検索系のテストは「テスト用の再実装」ではなく**本番経路**（`search_reader` 等）を通すこと。
CLI 全体の挙動（引数・終了コード・色/並列など）は `tests/cli.rs` の統合テストで担保する。
色に依存しない統合テストは `--color never` を明示してプラットフォーム差を避ける。

## 重要な不変条件・設計判断

これらは過去のレビューで固めた挙動。変更時は意図を確認すること。

- **巨大ファイル/行への耐性**: `MAX_LINE_BYTES` を超える 1 行はスキップ。先頭に NUL を
  含むファイルはバイナリ扱いでスキップ。`--max-filesize`（既定 `1M`、`0`=無制限）で上限。
- **エラー方針**:
  - 1 ファイルの**読み取り**失敗（`SearchError::Read`）は警告して走査継続。
  - **書き込み**失敗（`SearchError::Write`）は結果が不完全になるため abort。ただし
    `BrokenPipe`（`... | head` 等）は静かに正常終了。
  - 要求された**ルート自体**が辿れない場合は fatal（走査前に検証）。
- **色出力**: `anstyle`/`anstream` を使用。`--color auto` は `NO_COLOR`/`CLICOLOR`/dumb 端末を
  尊重し、Windows では VT を有効化。色 OFF 時はスタイルを出さず、内容中の ANSI も壊さない。
- **バッファリング**: 端末では std の行バッファに任せて即時表示。非端末（パイプ/リダイレクト）
  では `BufWriter` でスループット優先。
- **切り詰めと強調**: `--max-columns` は文字数ベース。可視範囲を先に決め、その範囲内の
  一致のみ強調する（幅ゼロ一致はスキップ）。
- **並列実行**: `-j/--threads`（0=自動=論理コア数、1=逐次）。`-j 1` は即時・決定的に出力。
  2 以上では候補ファイルを `par_bridge` でストリーム配分し（全候補を Vec に溜めないので
  初回出力が早く、`... | head` の早期終了も効く）、各ファイルを `ChunkWriter` 経由で共有
  writer へストリーム書き込みする（しきい値 `CHUNK_BYTES` を超えたら「最後の改行まで」を
  排他書き込み）。全マッチを溜めないのでメモリは並列数×チャンクサイズ程度に収まる。
  読み取りエラー時もバッファ済みの一致行は flush してから警告する（逐次経路と欠落差を
  作らない）。出力の小さいファイルはまとまって出るが、大きいファイルはチャンク分割され
  他ファイルと行が交互に出ることがある（各行にパス接頭辞があり判別可能）。順序は非決定的。
  書き込み先は並列共有のため `Send`（`io::stdout()` ベース）にしている。

## Git / PR 運用

- **`master` に直接コミットしない**。`feat/...`・`docs/...`・`fix/...` 等のブランチを切って
  PR を出す。
- コミットメッセージ・PR 説明は日本語（言語ポリシー参照）。1 コミット 1 関心事。
- コミットメッセージ末尾に以下を付ける:

  ```
  Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
  ```

- `gh`（GitHub CLI）で PR を作成・操作する。
- `/tmp`・`/target` は追跡しない（`.gitignore` 済み）。生成物をコミットしないよう
  `git add -A` 時は対象を確認する。

## やらないこと / 今後の候補

- 本番依存を安易に増やさない（軽量な単一バイナリを維持）。`--json` 実装時の `serde`/`serde_json`
  など、必要になった時点で追加する。CLI 統合テスト用の `assert_cmd`/`predicates`/`tempfile` は
  `[dev-dependencies]`（配布バイナリには含まれない）。
- 次の機能候補: `--json` / `--count` / `--files-with-matches`、
  `unicode-width` による表示幅ベースの切り詰め、並列時の決定的順序オプション（`--sort` 等）。
